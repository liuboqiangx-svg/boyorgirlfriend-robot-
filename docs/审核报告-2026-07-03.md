## 🔴 严重问题（安全/核心功能）

### 问题 1: 无用户认证系统，任何人可以"伪装"任何用户

- **位置**：全局 / 认证层
- **现象**：用户打开网站直接进入主界面，无需登录。通过设备ID（deviceId）识别用户，但该ID可被用户自行设置或伪造。
- **风险等级**：🔴 P0
- **复现步骤**：
  1. 打开浏览器开发者工具
  2. 在Console执行：`localStorage.setItem('deviceId', '任意其他用户的ID')`
  3. 刷新页面，即可"变身"为该用户
- **证据**：
  ```
  localStorage: {"deviceId":"user-iq6j7l3k3"}
  ```
  API请求头携带 `x-device-id: user-iq6j7l3k3`
- **影响评估**：
  - 任何用户都可以读取、发送其他用户的聊天记录
  - 隐私完全依赖"没人会猜ID"的掩耳盗铃策略
  - 如果ID格式被破解（如user-后跟规律数字），则可批量遍历用户数据
- **建议修复**：

  ⚠️ **注意**：加登录只是第一步，关键是怎么存 token。单纯在前端 localStorage 存 JWT 一样能被偷走。

  ```javascript
  // 方案A：服务端 Session + HttpOnly Cookie（推荐）
  // token 存在服务端，浏览器只能自动发送，看不到也改不了
  res.cookie('session_id', sessionId, {
    httpOnly: true,  // 前端 JS 看不到
    secure: true,    // 只能 HTTPS 传输
    sameSite: 'strict'  // 防止 CSRF
  })

  // 方案B：JWT + 服务端黑名单
  // 每次请求服务端验证 token，不只靠前端存储
  // 支持 token 主动失效（换密码/异地登录时）

  // 方案C：deviceId 签名（最低成本）
  // 后端用密钥对 deviceId 签名，前端改 deviceId 签名就对不上
  const signature = crypto.createHmac('sha256', SECRET_KEY)
    .update(deviceId)
    .digest('hex')
  ```

  **核心原则**：用户证明身份的凭证不能让用户自己能看到和修改

### 问题 2: API响应暴露内部数据库结构

- **位置**：/api/chat、/api/init 响应
- **现象**：API返回完整的内部数据结构，包括：
  - UUID格式的消息ID（如 `id: "eab44e0d-0937-4cfb-903f-031052a1d966"`）
  - 数据库字段如 `is_read`, `updated_at`, `created_at`
  - 内部状态如 `next_proactive_at`（下次主动消息时间）
- **风险等级**：🔴 P0
- **复现步骤**：
  1. 打开浏览器开发者工具 → Network
  2. 任意发送一条消息
  3. 查看POST /api/chat的响应
- **证据**：API响应包含完整的state对象：
  ```json
  {
    "state": {
      "id": "7047e24d-1e52-4169-8593-c417f270d6a6",
      "user_id": "user-iq6j7l3k3",
      "mood": "calm",
      "intimacy": 14,
      "mood_intensity": 50,
      "next_proactive_at": "2026-07-03T03:39:11.505Z"
    }
  }
  ```
- **影响评估**：
  - 攻击者可通过API了解数据库表结构
  - `next_proactive_at` 泄露了定时任务的实现细节
  - 内部ID暴露可能被用于后续攻击
- **建议修复**：
  ```javascript
  // 后端只返回必要字段，过滤内部实现细节
  const safeResponse = {
    mood: state.mood,
    moodLabel: state.moodLabel,
    intimacy: state.intimacy
    // 删除: id, user_id, next_proactive_at, updated_at 等
  }
  ```

### 问题 3: 后端缺少消息内容长度验证

- **位置**：前端输入框 + 后端/api/chat
- **现象**：前端通过禁用按钮阻止超长文本发送，但后端没有长度校验。恶意用户可以直接调用API发送任意长度内容。
- **风险等级**：🟡 P1
- **复现步骤**：
  ```bash
  curl -X POST https://boyorgirlfriend-robot.vercel.app/api/chat \
    -H "Content-Type: application/json" \
    -d '{"content":"非常非常长的内容...（10万字）","characterId":"lu-chen-001"}'
  ```
- **证据**：前端textbox在文本过长时按钮自动禁用，但API无校验
- **影响评估**：
  - 资源消耗攻击（发送超长消息耗尽AI token）
  - 数据库存储浪费
  - 可能绕过内容审核（超长文本中嵌入违规内容）
- **建议修复**：
  ```javascript
  // 后端添加验证
  if (content.length > 2000) {
    return res.status(400).json({ error: '消息长度不能超过2000字符' });
  }
  ```

## 🟡 中等问题

### 问题 4: 快捷回复展开后交互异常

- **位置**：主页底部快捷回复区域
- **现象**：点击快捷回复按钮后，按钮文字从"收起"变为"➕ 快捷回复"，再次点击应该是"添加"还是"展开"？用户无法收起已展开的快捷回复列表。
- **风险等级**：🟡 P1
- **复现步骤**：
  1. 默认状态显示"快捷回复:" + 6个快捷按钮 + "收起"按钮
  2. 点击"收起"后，按钮变为"快捷回复已展开"（disabled）
  3. 再次点击"☀️ 早安"等快捷按钮
  4. 按钮文字变成"➕ 快捷回复"
  5. 点击"➕ 快捷回复"后列表展开，但没有任何收起的方式
- **证据**：截图 `docs/audit-main-page.png`
- **影响评估**：用户被锁定在展开状态，无法恢复默认视图
- **建议修复**：
  ```jsx
  // 添加明确的展开/收起状态切换
  <button onClick={() => setExpanded(!expanded)}>
    {expanded ? '➖ 收起快捷回复' : '➕ 添加快捷回复'}
  </button>
  ```
